# TOR (Terms of Reference) ฉบับสมบูรณ์

> เอกสารนี้จัดทำเพื่อทดแทนร่างเดิม และแก้ไขจุดบกพร่องที่พบบ่อยใน TOR ได้แก่ ขอบเขตงานไม่ชัดเจน, เกณฑ์รับมอบไม่วัดผลได้, และเงื่อนไขสัญญาไม่ครบถ้วน

## 1) หลักการและเหตุผล
หน่วยงานมีความจำเป็นต้องจัดหา/พัฒนาระบบเพื่อรองรับการปฏิบัติงานอย่างมีประสิทธิภาพ โปร่งใส ตรวจสอบได้ และลดขั้นตอนการดำเนินงานที่ซ้ำซ้อน

## 2) วัตถุประสงค์
1. เพื่อจัดหา/พัฒนาระบบให้ใช้งานได้จริงตามภารกิจของหน่วยงาน
2. เพื่อเพิ่มความถูกต้องของข้อมูลและลดระยะเวลาการปฏิบัติงาน
3. เพื่อให้สามารถติดตามผลและออกรายงานได้ตามตัวชี้วัด

## 3) ขอบเขตงาน (Scope of Work)
### 3.1 งานวิเคราะห์และออกแบบ
- เก็บความต้องการ (Requirement Gathering) จากผู้เกี่ยวข้อง
- จัดทำเอกสาร SRS / Use Case / Process Flow
- ออกแบบสถาปัตยกรรมระบบและความปลอดภัย

### 3.2 งานพัฒนา/ติดตั้ง
- พัฒนาระบบตามคุณลักษณะขั้นต่ำที่กำหนด
- ติดตั้งในสภาพแวดล้อมที่หน่วยงานกำหนด (DEV/UAT/PROD)
- จัดทำระบบสำรองข้อมูลและแผนกู้คืน

### 3.3 งานทดสอบและส่งมอบ
- ทดสอบหน่วย (Unit Test), รวมระบบ (Integration Test), และยอมรับผู้ใช้ (UAT)
- แก้ไขข้อบกพร่องก่อนส่งมอบทุกประเด็นระดับ Critical/High
- ส่งมอบซอร์สโค้ด/ไฟล์ติดตั้ง/คู่มือ/เอกสารทั้งหมด

### 3.4 งานฝึกอบรมและถ่ายทอดความรู้
- อบรมผู้ดูแลระบบและผู้ใช้งาน
- จัดทำคู่มือ Admin และ User Manual ภาษาไทย
- ส่งมอบวิดีโอ/สไลด์ประกอบการอบรม (ถ้ามี)

## 4) คุณลักษณะขั้นต่ำ (Minimum Requirements)
1. รองรับผู้ใช้งานพร้อมกันไม่น้อยกว่า [xx] คน
2. มีระบบสิทธิ์ผู้ใช้งานแบบ Role-based Access Control (RBAC)
3. บันทึกประวัติการใช้งาน (Audit Log) ตรวจสอบย้อนหลังได้ไม่น้อยกว่า [xx] วัน
4. รองรับการสำรองข้อมูลอัตโนมัติอย่างน้อยวันละ 1 ครั้ง
5. รองรับการเชื่อมต่อผ่าน HTTPS/TLS และจัดการรหัสผ่านตามมาตรฐานความปลอดภัย
6. รองรับการออกรายงานในรูปแบบ PDF/Excel

## 5) มาตรฐานและความปลอดภัย
- ปฏิบัติตาม พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล (PDPA) และกฎหมายที่เกี่ยวข้อง
- มีการควบคุมสิทธิ์เข้าถึงข้อมูลตามหลัก Least Privilege
- มีมาตรการป้องกันช่องโหว่พื้นฐาน เช่น SQL Injection, XSS, CSRF
- ส่งมอบรายงานผลการทดสอบความปลอดภัย (ถ้ามีข้อกำหนด)

## 6) ระยะเวลาดำเนินการ
- ระยะเวลาดำเนินโครงการรวมไม่เกิน [xx] วัน นับถัดจากวันลงนามสัญญา
- แผนงานหลัก:
  - งวดที่ 1: วิเคราะห์/ออกแบบ
  - งวดที่ 2: พัฒนาและทดสอบภายใน
  - งวดที่ 3: UAT แก้ไข และส่งมอบ

## 7) การส่งมอบงานและเกณฑ์การตรวจรับ
### 7.1 เอกสาร/ผลลัพธ์ที่ต้องส่งมอบ
- เอกสารวิเคราะห์ความต้องการ (SRS)
- เอกสารออกแบบระบบ
- แผนและผลการทดสอบ
- คู่มือผู้ใช้/ผู้ดูแล
- ซอร์สโค้ดและทรัพย์สินดิจิทัลที่เกี่ยวข้อง

### 7.2 เกณฑ์ผ่านการตรวจรับ (Acceptance Criteria)
1. ฟังก์ชันครบตามขอบเขตงานข้อ 3
2. ข้อบกพร่องระดับ Critical/High = 0 รายการ
3. ข้อบกพร่องระดับ Medium/Low ต้องมีแผนแก้ไขพร้อมกำหนดเวลา
4. ผล UAT ผ่านไม่น้อยกว่า [xx]% ของ Test Cases ทั้งหมด
5. เอกสารส่งมอบครบถ้วนและใช้งานได้จริง

## 8) การรับประกันผลงาน
- รับประกันระบบไม่น้อยกว่า [xx] เดือนนับจากวันตรวจรับงวดสุดท้าย
- ระยะเวลาเริ่มดำเนินการแก้ไขปัญหา (Response Time): ไม่เกิน [x] ชั่วโมงทำการ
- ระยะเวลาแก้ไขแล้วเสร็จ (Resolution Time) ตามระดับความรุนแรงที่ตกลงร่วมกัน

## 9) เงื่อนไขการชำระเงิน
- ชำระเป็นงวดตามผลสำเร็จของงานและการตรวจรับ
- แต่ละงวดต้องมีรายงานความก้าวหน้าและหลักฐานการส่งมอบ
- หน่วยงานมีสิทธิ์ชะลอการจ่ายหากงานไม่เป็นไปตาม TOR

## 10) ค่าปรับและการบอกเลิกสัญญา
- หากส่งมอบล่าช้า ให้คิดค่าปรับตามอัตราที่ระเบียบราชการกำหนด
- หากไม่สามารถส่งมอบงานตามสาระสำคัญของสัญญา หน่วยงานมีสิทธิบอกเลิกสัญญา

## 11) สิทธิในทรัพย์สินทางปัญญา
- ผลงาน เอกสาร ซอร์สโค้ด และข้อมูลที่พัฒนาภายใต้โครงการนี้ เป็นสิทธิของหน่วยงาน
- ผู้รับจ้างต้องไม่เปิดเผยข้อมูลของหน่วยงานโดยไม่ได้รับอนุญาต

## 12) ข้อกำหนดเพิ่มเติม
- ผู้เสนอราคาต้องมีผลงานอ้างอิงที่เกี่ยวข้องอย่างน้อย [x] โครงการ
- ผู้จัดการโครงการ/ทีมงานหลักต้องมีประสบการณ์ตามที่กำหนด
- หน่วยงานสงวนสิทธิ์ปรับรายละเอียดที่ไม่กระทบสาระสำคัญของโครงการ

## ภาคผนวก A: ตาราง SLA ตัวอย่าง
| ระดับเหตุขัดข้อง | ตัวอย่างอาการ | Response Time | Resolution Time |
|---|---|---|---|
| Critical | ระบบล่มใช้งานไม่ได้ทั้งระบบ | ≤ 1 ชม. | ≤ 8 ชม. |
| High | ฟังก์ชันหลักใช้งานไม่ได้บางส่วน | ≤ 4 ชม. | ≤ 1 วันทำการ |
| Medium | มีผลกระทบแต่มีทางเลี่ยงได้ | ≤ 1 วันทำการ | ≤ 3 วันทำการ |
| Low | ปรับปรุงทั่วไป/ความสวยงาม | ≤ 2 วันทำการ | ตามรอบบำรุงรักษา |

## ภาคผนวก B: Checklist ปิดบัคก่อนส่งมอบ
- [ ] ไม่มีบัคระดับ Critical/High คงค้าง
- [ ] UAT ผ่านตามเกณฑ์
- [ ] เอกสารอัปเดตตรงกับระบบจริง
- [ ] ส่งมอบซอร์สโค้ดและสิทธิ์เข้าถึงที่จำเป็นครบถ้วน
- [ ] มีแผนรองรับหลังขึ้นระบบจริง (Go-live Support)
